# 웹 테스트 실행 스킬

## 개요
Notion 데이터베이스의 테스트 케이스를 읽어 Chrome 브라우저에서 자동 테스트를 수행하고, 결과를 Notion에 AI 태그(통과(AI)/실패(AI)/보류(AI))로 기록하는 범용 웹 테스트 실행 스킬입니다.
`web-testcase-creator` 스킬로 생성된 테스트 케이스를 실행하는 것이 주 용도이며, 동일한 스키마를 사용하는 모든 Notion 테스트 DB에 대해 동작합니다.

## 인자
- `notion_db_url` (필수): Notion 테스트 케이스 데이터베이스 URL
- `server_url` (선택): 테스트 대상 서버 URL. 미지정 시 각 테스트 케이스의 "대상 URL" 속성 사용
- `username` (선택): 로그인 계정. 미지정 시 사용자에게 질문
- `password` (선택): 로그인 비밀번호. 미지정 시 사용자에게 질문

## 실행 절차

### 1단계: Notion 테스트 케이스 수집
1. `notion-fetch`로 `notion_db_url`의 데이터베이스 스키마 및 데이터소스 URL 확인
2. `notion-search`로 해당 데이터소스의 모든 테스트 케이스 페이지 검색
   - 여러 쿼리 사용: 모듈별 키워드, TC_ID 접두사별 검색
3. 각 테스트 케이스 페이지를 `notion-fetch`로 상세 조회하여 수집:
   - TC_ID, 모듈, 화면/메뉴, 테스트 시나리오, 테스트 유형
   - 사전조건, 자동화 여부, 현재 테스트 결과
   - 대상 URL (있는 경우)
   - 페이지 본문: 테스트 절차, 기대결과, UI 요소 힌트
4. TC_ID 기준 오름차순 정렬

### 2단계: 실행 대상 필터링 및 순서 결정

#### 2-1. 실행 대상 필터링
- `자동화` 필드가 "Yes" 또는 "Partial"인 케이스만 실행 대상
- `자동화`가 "No"인 케이스는 건너뛰고 보류(AI) 처리
- 이미 "통과(AI)"인 케이스는 사용자에게 재실행 여부 확인

#### 2-2. 사전조건 의존성 분석
각 테스트의 사전조건을 파싱하여 실행 순서 결정:
```
1. 사전조건에서 TC_ID 참조 추출 (예: "XXX-LOGIN-001 선행")
2. 의존성 그래프 구성
3. 위상 정렬(topological sort)로 실행 순서 결정
4. 순환 의존성 발견 시 사용자에게 알림
```

#### 2-3. 실행 계획 사용자 확인
```
실행할 테스트 케이스: N건 (전체 M건 중)
건너뛸 케이스: K건 (자동화: No)

실행 순서:
1. XXX-LOGIN-001: 로그인 기능 확인
2. XXX-DASH-001: 대시보드 로드 확인
3. XXX-SET-001: 설정 화면 로드
...

진행할까요?
```

### 3단계: Chrome 브라우저 테스트 실행

#### 3-1. 브라우저 초기화
```
1. tabs_context_mcp으로 탭 컨텍스트 확인 (createIfEmpty: true)
2. 새 탭 생성 (tabs_create_mcp)
3. server_url (또는 테스트 케이스의 대상 URL)로 navigate
4. SSL 인증서 경고가 나타나면:
   - JavaScript로 #details-button 클릭
   - #proceed-link 클릭하여 진행
5. 페이지 로드 대기 (wait 3초 + screenshot)
```

#### 3-2. 로그인 처리
```
1. screenshot으로 현재 화면 확인
2. 로그인 페이지인 경우:
   a. username/password가 인자로 제공되었으면 사용
   b. 미제공 시 AskUserQuestion으로 사용자에게 요청
   c. find로 계정/이메일/아이디 입력 필드 → form_input
   d. find로 비밀번호 입력 필드 → form_input
   e. find로 로그인/Login/Sign in 버튼 → 클릭
   f. 로그인 성공 확인 (URL 변경 + 대시보드/메인 로드)
3. 로그인 불필요 시 바로 테스트 시작
```

#### 3-3. 각 테스트 케이스 실행
각 테스트 케이스에 대해 다음 프로세스를 실행합니다:

```
1. 테스트 절차 파싱 및 실행:
   각 단계를 아래 패턴에 매칭하여 브라우저 동작 수행:

   [네비게이션]
   - "사이드바에서 '[메뉴]' 클릭" → find('[메뉴]') → click
   - "'[메뉴]' 하위 메뉴에서 '[항목]' 클릭" → find('[메뉴]') → click → find('[항목]') → click
   - "[URL]로 이동" → navigate(URL)
   - "페이지 로드 대기/확인" → wait 2초 → screenshot

   [입력]
   - "'[라벨]' 필드에 '[값]' 입력" → find('[라벨]') → form_input('[값]')
   - "'[라벨]' 드롭다운에서 '[옵션]' 선택" → find('[라벨]') → click → find('[옵션]') → click
   - "'[라벨]' 체크박스 체크/해제" → find('[라벨]') → click
   - "'[라벨]' 토글 ON/OFF" → find('[라벨]') → click

   [클릭]
   - "'[텍스트]' 버튼 클릭" → find('[텍스트]') → click
   - "[+] 버튼 클릭" → find('추가' 또는 '+') → click
   - "확인 대화상자에서 '확인' 클릭" → find('확인' 또는 'OK') → click

   [확인]
   - "'[텍스트]' 존재/표시 확인" → find('[텍스트]') 또는 get_page_text로 확인
   - "테이블에 '[값]' 행 존재 확인" → get_page_text로 텍스트 검색
   - "URL이 '[패턴]'인지 확인" → javascript_tool로 location.href 확인
   - "에러 메시지 없음 확인" → screenshot + read_page로 에러 요소 탐색

2. UI 요소 힌트 활용:
   - find 도구의 쿼리로 "UI 요소 힌트" 섹션의 텍스트 사용
   - 기본 find 실패 시 힌트의 대체 텍스트로 재시도

3. 기대결과 검증:
   a. screenshot으로 결과 화면 캡처
   b. read_page 또는 get_page_text로 텍스트 내용 확인
   c. 기대결과 항목별 충족 여부 판정

4. 결과 판정:
   - 통과(AI): 모든 기대결과 충족
   - 실패(AI): 기대결과 미충족 또는 에러 발생
   - 보류(AI): 사전조건 미충족, 환경 제약, 수동 확인 필요, 또는 자동화=No
```

### 4단계: Notion 결과 기록
각 테스트 케이스 완료 후 **즉시** 결과를 기록합니다 (배치 처리 금지):
```
1. notion-update-page로 테스트 결과 업데이트:
   - command: "update_properties"
   - page_id: 해당 테스트 케이스의 Notion 페이지 ID
   - properties: {"테스트 결과": "통과(AI)" | "실패(AI)" | "보류(AI)"}
```

### 5단계: 결과 보고
모든 테스트 완료 후 사용자에게 종합 결과 보고:
```
테스트 실행 완료:
- 총 실행: N건 / 전체 M건
- 통과(AI): N건
- 실패(AI): N건
- 보류(AI): N건
- 건너뜀 (자동화 No): N건

실패 케이스:
- [TC_ID]: [시나리오] - [실패 사유 요약]
- [TC_ID]: [시나리오] - [실패 사유 요약]

보류 케이스:
- [TC_ID]: [시나리오] - [보류 사유]
```

## 에러 처리
- **페이지 로드 실패**: 3초 대기 후 재시도 (최대 3회), 실패 시 보류(AI)
- **요소 찾기 실패**: UI 요소 힌트의 대체 텍스트로 재시도, find 쿼리를 다른 표현으로 변경하여 재시도, 그래도 실패 시 보류(AI)
- **SSL 인증서 경고**: JavaScript로 자동 우회
- **세션 만료**: 로그인 페이지 감지 시 재로그인 후 해당 테스트부터 재시작
- **서버 연결 불가**: 사용자에게 알림 후 대기
- **예상치 못한 팝업/모달**: screenshot 확인 후 ESC 또는 닫기 버튼으로 해제
- **Notion API 오류**: 재시도 후 실패 시 로컬에 결과 기록하고 사용자에게 알림

## 주의사항
- 테스트 실행 전 각 테스트의 사전조건 충족 여부를 확인하고, 미충족 시 선행 테스트를 먼저 실행
- 데이터 변경 테스트(생성/수정/삭제) 실행 후에는 원래 상태로 복원 시도 (테스트 데이터 정리)
- 파일 업로드가 필요한 테스트는 보류(AI) 처리
- 외부 서버 연동 테스트(SMTP, LDAP 등)는 환경에 따라 보류(AI) 처리 가능
- 각 테스트 완료 후 즉시 Notion에 결과 기록 (중간에 중단되어도 결과가 유실되지 않도록)
- server_url과 테스트 케이스의 "대상 URL"이 다른 경우, server_url 인자가 우선
