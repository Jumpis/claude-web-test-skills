# 웹 테스트 케이스 생성 스킬

## 개요
`nav-analyze` 스킬로 생성된 네비게이션 구조 MD 파일을 읽고, 사용자가 지정한 특정 피쳐/화면에 대한 테스트 케이스를 설계하여 Notion 데이터베이스에 생성합니다.
생성된 테스트 케이스는 `web-test-runner` 스킬로 자동 실행할 수 있습니다.

## 인자
- `nav_md_path` (필수): `nav-analyze` 스킬로 생성된 네비게이션 구조 MD 파일 경로
- `notion_db_url` (필수): 테스트 케이스를 기록할 Notion 데이터베이스 URL

## 실행 절차

### 1단계: 네비게이션 구조 파일 로드
1. Read 도구로 `nav_md_path` 파일 읽기
2. 파일 내용 파싱:
   - 대상 URL 추출
   - GNB 메뉴 구조 파싱
   - LNB 메뉴 구조 파싱
   - 화면별 상세 분석 정보 파싱 (URL, 화면 유형, 주요 기능, UI 요소 힌트)
3. 파일이 존재하지 않거나 형식이 다른 경우 사용자에게 안내:
   - "`/nav-analyze [url]` 명령으로 먼저 네비게이션 구조를 분석해주세요."

### 2단계: 사용자 입력 수집
AskUserQuestion 도구로 테스트 대상을 확인합니다.

**질문 1: 테스트 대상 선택**
MD 파일에서 파싱한 메뉴 목록을 보여주고 선택하게 합니다:
```
네비게이션 구조에서 다음 화면들이 발견되었습니다:
1. 대시보드 > 상태 모니터링
2. 설정 > 사용자 관리
3. 설정 > 권한 관리
...

어떤 화면/피쳐에 대해 테스트 케이스를 생성할까요?
(번호, 메뉴 이름, 또는 직접 입력)
```

**질문 2: 테스트 범위 (선택한 화면에 대해)**
```
선택한 화면에 대해 어떤 범위의 테스트를 생성할까요?
- 전체: 해당 화면의 모든 기능에 대한 테스트 (CRUD, UI, 검증 등)
- 특정 기능: 사용자가 지정한 특정 기능만
- 사용자 정의: 사용자가 직접 절차/기대결과 제공
```

**질문 3 (특정 기능/사용자 정의 선택 시):**
- 특정 기능: MD에서 해당 화면의 "주요 기능" 목록을 보여주고 선택
- 사용자 정의: 테스트 절차, 사전조건, 기대결과, UI 힌트를 자유 형식으로 입력받기

### 3단계: Notion 데이터베이스 준비
1. `notion-fetch`로 `notion_db_url` 스키마 확인
2. 필수 속성 확인 및 누락 시 자동 추가 (`notion-update-data-source`):

| 속성명 | 타입 | 설명 |
|--------|------|------|
| TC_ID | TITLE | 테스트 케이스 고유 ID |
| 모듈 | SELECT | GNB/LNB 1차 메뉴명 |
| 화면/메뉴 | RICH_TEXT | 전체 메뉴 경로 |
| 테스트 시나리오 | RICH_TEXT | 한 줄 요약 |
| 테스트 유형 | SELECT | 기능/UI/보안/성능 |
| 사전조건 | RICH_TEXT | 선행 조건 |
| 자동화 | SELECT | Yes/Partial/No |
| 테스트 결과 | SELECT | 미실행/통과(AI)/실패(AI)/보류(AI) |
| 대상 URL | URL | 테스트 대상 웹 애플리케이션 URL |

3. 기존 테스트 케이스 조회 (`notion-search`)하여 TC_ID 목록 수집

### 4단계: TC_ID 채번
MD 파일의 대상 URL에서 프로젝트 접두사를 자동 생성합니다:
```
규칙: [PREFIX]-[MODULE]-[NUMBER]

PREFIX 결정:
- URL의 도메인/경로에서 프로젝트 식별자 추출
  예: admin.example.com → ADM
  예: cms.company.co.kr → CMS
  예: 192.168.1.100:8080/portal → PTL
- 사용자에게 PREFIX를 확인 또는 변경 요청

MODULE 결정:
- 1차 메뉴명의 약어 자동 생성
  예: 설정 → SET, 대시보드 → DASH, 사용자 관리 → USER
  예: Settings → SET, Dashboard → DASH, Users → USER
- 기존 TC_ID에서 이미 사용된 MODULE 약어가 있으면 동일하게 사용

NUMBER: 해당 모듈 내 001부터 순차 증가 (기존 최대 번호 + 1)
```

### 5단계: 테스트 케이스 설계

#### 5-1. 전체 범위 선택 시 자동 생성 규칙
MD 파일의 화면 분석 정보를 기반으로 화면 유형별 테스트 세트를 자동 설계:

**목록(List) 화면:**
| 테스트 시나리오 | 테스트 유형 | 자동화 |
|----------------|------------|--------|
| 화면 정상 로드 확인 | UI | Yes |
| 목록 데이터 표시 확인 | 기능 | Yes |
| 검색 기능 동작 확인 | 기능 | Yes |
| 필터 기능 동작 확인 | 기능 | Yes |
| 페이지네이션 동작 확인 | 기능 | Yes |
| 신규 항목 생성 | 기능 | Yes |
| 항목 상세 조회 | 기능 | Yes |
| 항목 수정 | 기능 | Yes |
| 항목 삭제 | 기능 | Yes |
| 필수 입력 누락 시 검증 | 기능 | Yes |

**설정(Settings) 화면:**
| 테스트 시나리오 | 테스트 유형 | 자동화 |
|----------------|------------|--------|
| 설정 화면 정상 로드 | UI | Yes |
| 현재 설정값 표시 확인 | 기능 | Yes |
| 설정값 변경 및 저장 | 기능 | Yes |
| 토글 설정 ON/OFF 전환 | 기능 | Yes |
| 잘못된 값 입력 시 검증 | 기능 | Yes |

**대시보드(Dashboard) 화면:**
| 테스트 시나리오 | 테스트 유형 | 자동화 |
|----------------|------------|--------|
| 대시보드 정상 로드 | UI | Yes |
| 위젯/차트 표시 확인 | UI | Partial |
| 데이터 갱신 확인 | 기능 | Partial |

**폼(Form) 화면:**
| 테스트 시나리오 | 테스트 유형 | 자동화 |
|----------------|------------|--------|
| 폼 화면 정상 로드 | UI | Yes |
| 정상 데이터 입력 후 제출 | 기능 | Yes |
| 필수 입력 누락 시 검증 | 기능 | Yes |
| 잘못된 형식 입력 시 검증 | 기능 | Yes |
| 취소 버튼 동작 확인 | 기능 | Yes |

#### 5-2. 테스트 절차 상세화
각 테스트 케이스의 절차를 `web-test-runner` 스킬이 실행할 수 있도록 구체적으로 작성합니다.
MD 파일의 UI 요소 힌트를 활용하여 각 단계에 대응하는 브라우저 동작을 명시:

```
테스트 절차 작성 규칙:
1. 네비게이션: "사이드바에서 '[메뉴명]' 클릭" 또는 "[URL]로 이동"
2. 대기: "페이지 로드 대기" (runner가 자동으로 wait + screenshot)
3. 확인: "[요소] 존재/표시 여부 확인" (runner가 find/read_page로 확인)
4. 입력: "'[라벨]' 필드에 '[값]' 입력" (runner가 find + form_input)
5. 클릭: "'[텍스트]' 버튼 클릭" (runner가 find + click)
6. 검증: "[기대 텍스트/상태] 확인" (runner가 screenshot + read_page/get_page_text)
```

#### 5-3. 사전조건 연결
- 로그인 필요 → "로그인 상태 ([PREFIX]-LOGIN-001 선행)"
- 데이터 의존 → 해당 생성 테스트의 TC_ID 참조
- 특정 상태 필요 → 상태를 만드는 테스트의 TC_ID 참조
- 기존 TC_ID가 없는 경우 텍스트로 기술

#### 5-4. 기대결과 작성
```
기대결과 작성 규칙:
- 화면 로드: "페이지 타이틀/헤더에 '[텍스트]' 표시" 또는 "URL이 '[패턴]'으로 변경"
- CRUD 성공: "성공 메시지/토스트 표시" + "목록에 항목 반영"
- 검증 실패: "에러 메시지 '[텍스트]' 표시" 또는 "필드에 빨간 테두리/경고"
- 상태 변경: "토글/버튼 상태가 [ON/OFF]로 변경"
```

### 6단계: 사용자 확인
생성할 테스트 케이스 전체 목록을 사용자에게 보여줍니다:

```
[PREFIX]-[MODULE]-001: 화면 정상 로드 확인 (UI, Yes)
[PREFIX]-[MODULE]-002: 목록 데이터 표시 확인 (기능, Yes)
[PREFIX]-[MODULE]-003: 신규 항목 생성 (기능, Yes)
...

총 N건의 테스트 케이스를 생성합니다. 진행할까요?
수정/추가/삭제할 항목이 있으면 알려주세요.
```

사용자가 수정 요청 시 반영 후 다시 확인합니다.

### 7단계: Notion에 테스트 케이스 생성
사용자 확인 후:
1. `notion-create-pages`로 테스트 케이스 페이지 생성
   - parent: data_source_id (3단계에서 확인)
   - 한 번에 최대 10개씩 배치 생성
2. 각 페이지의 properties:
```json
{
  "TC_ID": "[PREFIX]-[MODULE]-[NUMBER]",
  "모듈": "[1차 메뉴명]",
  "화면/메뉴": "[전체 메뉴 경로]",
  "테스트 시나리오": "[한 줄 요약]",
  "테스트 유형": "[기능/UI/보안/성능]",
  "사전조건": "[선행 조건]",
  "자동화": "[Yes/Partial/No]",
  "테스트 결과": "미실행",
  "대상 URL": "[url from MD]"
}
```
3. 각 페이지의 content (본문):
```markdown
## 테스트 절차
1. [단계 1]
2. [단계 2]
...

## 기대결과
- [검증 항목 1]
- [검증 항목 2]
...

## UI 요소 힌트
- [요소]: [find 쿼리 힌트]
...
```

### 8단계: 결과 보고
```
생성 완료:
- 총 생성: N건
- 모듈별: [모듈A] N건, [모듈B] N건
- 자동화: Yes N건, Partial N건, No N건
- Notion DB: [url]

`/web-test [notion_db_url] [server_url]` 명령으로 테스트를 실행할 수 있습니다.
```

## 에러 처리
- **MD 파일 없음**: `/nav-analyze` 먼저 실행하도록 안내
- **MD 파일 형식 불일치**: 파싱 가능한 범위에서 최대한 추출, 부족한 정보는 사용자에게 질문
- **Notion DB 접근 불가**: URL 확인 요청
- **스키마 불일치**: 누락 속성 자동 추가 후 사용자에게 알림
- **TC_ID 중복**: 자동으로 다음 번호 부여
- **Notion API 오류**: 재시도 후 실패 시 사용자에게 알림

## 주의사항
- MD 파일의 UI 요소 힌트를 테스트 절차와 본문에 적극 반영
- 테스트 절차는 `web-test-runner` 스킬이 브라우저 자동화로 실행할 수 있는 수준으로 구체적으로 작성
- 사전조건의 TC_ID 참조는 기존 테스트 케이스와 대조하여 정확성 확보
- 사용자가 제공한 추가 정보(절차, 기대결과 등)는 자동 생성 내용보다 우선
- 생성 전 반드시 사용자 확인을 받을 것
